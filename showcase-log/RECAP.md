# Recap Generation Flow

Reference doc for building a recap page — read this when a recap is actually requested
(see SKILL.md's on-demand trigger phrases), not as part of routine setup or per-entry
logging.

The recap page has two layers — a **deterministic base** generated by `generate-recap.mjs`,
and **AI-analyzed sections** that Claude fills in afterwards. Each run writes a dated file,
`session-log/YYYY-MM-DD-Recap.html`, so recaps accumulate side by side instead of one
overwriting the last — regenerating later the same day overwrites that day's file, but a
recap from a different day is a separate file. The page itself is a single warm-toned card
(JetBrains Mono for headers/data, IBM Plex Sans for body text — both loaded from Google
Fonts with a system-font fallback), mobile-friendly, and renders without JavaScript (day
drill-downs work via a `:target` CSS fallback). JS is used only for the slide-out day panel
and any interactive enhancement Claude adds to Workstreams.

### Step 1: Scope, then (maybe) section picker

When the user asks "make a recap" (or equivalent), ask via AskUserQuestion — `multiSelect:
false`, header `"Recap scope"`, question "Want the complete report, or pick specific
sections?":

1. **"Complete report (Recommended)"** — *"All five sections: Daily activity with cost &
   time detail, Workstreams, Milestones & capabilities timeline, How you use Claude, and
   Findings."*
2. **"Deterministic only"** — *"Just daily activity with cost & time detail — generated by
   script. Instant, and uses minimal tokens."*
3. **"Choose specific sections"** — *"Pick exactly which of the five sections to
   include."*

Options 1 and 2 resolve the section list immediately (all five, or just the one
deterministic section) — skip straight to Step 2. Option 3 asks a second question, via
AskUserQuestion — `multiSelect: true`, header `"Sections"`, question "Which sections
should the recap include?":

1. **"Daily activity with cost & time detail"** (deterministic) — *"Cost bars per day (tap
   to see entries), plus cost-by-model and time-window tables."*
2. **"Workstreams"** (AI) — *"Gantt-style overview of classified activity threads by day,
   with an accordion detail list."*
3. **"Milestones & capabilities timeline"** (AI) — *"Dual timeline pairing declared
   milestones with capabilities shipped."*
4. **"How you use Claude"** (AI) — *"Patterns in how you prompt and collaborate with
   Claude."*
5. **"Findings, ranked by value"** (AI) — *"Ranked, actionable findings — project
   direction and how you could work with Claude better — closed by a 'three things this
   week' capstone."*

Nothing is pre-checked in this second question — `AskUserQuestion` has no way to
pre-select an option, so it's always a from-scratch pick regardless of what's "typical."

**Daily activity and Cost & time are always one bundled choice, never two** — there is no
way to include one without the other, in either question. They render as two separate
`<section>` elements in the template (different headings, different layouts) but share a
single `data-section="daily-activity-cost"` attribute for exactly this reason: one
inclusion/exclusion decision, applied to both at once (see Step 4).

### Step 2: Generate the base

If the "Daily activity with cost & time detail" section is included, run
`node _scripts/usage-snapshot.mjs` first (no `--auto`) so the recap's cost numbers reflect
today, not whatever the last debounced hook happened to catch. Then run
`node _scripts/generate-recap.mjs`. This creates
`session-log/YYYY-MM-DD-Recap.html` (today's date) with:
- All deterministic sections filled in (daily bars, cost tables)
- AI sections showing placeholder text ("This section is generated by Claude…")
- Pre-rendered static HTML for the daily-activity chart and day drill-downs (works in
  viewers that don't execute JS, via the `:target` fallback)
- Mobile-responsive CSS (breakpoints at 680px and 400px)
- Interactive JS for the slide-out day panel (progressive enhancement over the `:target` fallback)

### Step 3: Fill AI sections (if selected)

For each AI section the user selected, read the log entries and write the content directly
into `recap.html`, replacing the placeholder `<div>`. Follow these patterns:

#### Workstreams

1. **Read the cache, if present.** If `session-log/.workstreams.json` exists, read it. It
   holds two things: the workstream definitions (name, color, description) and a per-entry
   classification map keyed by each entry's start timestamp (`YYYY-MM-DD HH:MM`, taken
   straight from its `###` heading):
   ```json
   {
     "workstreams": [
       { "name": "Content corpus", "color": "var(--ws-1)", "description": "one-line description" }
     ],
     "entries": {
       "2026-06-10 14:32": "Content corpus"
     }
   }
   ```
   This is the **only** place workstream classification is persisted; don't also stash it
   as a comment in the recap or anywhere else, or which copy is authoritative becomes
   ambiguous and classifications will drift between runs.

2. **Classify only what's new.** Compare the cache's `entries` keys against every entry
   currently in the log (`session-log.md` + `archive/*.md`). Any entry whose start
   timestamp isn't already a key is new since the cache was last written — classify just
   those, into the *existing* workstream list; don't re-classify or re-confirm entries
   already in the cache, and don't re-run the whole-project pass described in step 3. On a
   project that's had several recaps already, re-classifying everything from scratch every
   time is wasted work and risks a given entry landing in a different workstream between
   runs for no reason. If a new entry doesn't fit any existing workstream well, propose
   adding one and confirm with the user rather than silently expanding the taxonomy or
   forcing a bad fit. Merge the new classifications into `entries` and write the file back.
   If there's nothing new (every current entry is already a key), skip straight to step 4
   with the cache as-is — nothing to write, nothing to ask.

3. **First time for this project (no cache yet):** read all log entries and auto-classify
   each into workstreams by analyzing the heading and content. Each workstream gets a name,
   color, and description. **Keep names short — 14 characters is a safe ceiling.** The
   `.ws-gantt-label` column is fixed-width to keep every row's percentage and swatch
   aligned; a name like "Review & distribution" or "Docs, audits & quality" overflows it
   and clips the percentage right off the row. Prefer "Distribution" and "Docs & quality"
   over their longer, more-precise-sounding equivalents — the accordion body and blurb are
   where the fuller description belongs.

   If the log is large (a long-running project with hundreds of entries across many
   archive files), classifying every entry in one pass can blow past a single context
   window. Fan the reads out instead: spawn one background agent per archive file (or a
   batch of files) to read and return a compact per-entry classification, then merge the
   results yourself rather than re-reading the raw entries. This keeps the classification
   step reliable regardless of project size, and avoids truncating history silently — a
   partial read that never mentions what it skipped will produce workstream numbers that
   look plausible but are wrong.

   Present the classified workstreams to the user and ask them to validate: "I classified
   your work into these N workstreams — does this look right? Any to rename, merge, or
   split?" Adjust based on feedback, then write `session-log/.workstreams.json` in the
   schema shown in step 1.
4. Generate the workstreams section as a **gantt-style overview** paired with an
   **accordion detail list** — not a positioned swimlane. Compute two numbers per
   workstream: a **default share** (that workstream's percentage of total activity across
   the *whole* project — these should sum to ~100 across all workstreams), and a
   **per-day share** for each active day (that workstream's percentage of activity *that
   day* — each day's shares across workstreams should also sum to ~100, using 0 for a
   workstream untouched that day). Both are real percentages the hover interaction swaps
   between — never leave either as a placeholder. **Use the same basis (entry count) for
   both.** Deriving the default from cost totals and the per-day figures from entry counts
   — or any other mixed basis — produces numbers that don't reconcile with each other, and
   a day with one entry can end up showing a suspiciously smooth six-way split instead of
   the 100%-one-workstream split its own entry list implies.

   Markup, one `.ws-gantt-row` per workstream:
   ```html
   <div class="hover-info">Hover a day to see the workstream mix; click through to that day's log.</div>
   <div class="ws-gantt-wrap"><div class="ws-gantt-inner" style="position:relative;">
     <div class="ws-gantt-row">
       <div class="ws-gantt-label">
         <span class="ws-gantt-pct" data-default="20" style="color:var(--ws-1);">20%</span>
         <span class="ws-gantt-name">Content corpus</span>
         <span class="ws-gantt-swatch" style="background:var(--ws-1);"></span>
       </div>
       <div class="ws-gantt-track">
         <div class="ws-gantt-cell" style="background:var(--ws-1);opacity:0.3" data-pct="15"></div>
         <!-- one .ws-gantt-cell per active day, in day order, matching the .ws-gantt-hit order below.
              opacity = 0.1 + (share/100)*0.8 so 0% is faint but never invisible.
              data-pct is that day's real share for this workstream — the hover script reads it. -->
       </div>
     </div>
     <!-- one .ws-gantt-row per workstream, same day order in every row's cells -->
     <div class="ws-gantt-hitlayer">
       <a class="ws-gantt-hit" href="#panel-2026-06-10" data-day="2026-06-10"></a>
       <!-- one .ws-gantt-hit per active day, same count/order as the day columns in every row -->
     </div>
   </div></div>
   <div class="ws-gantt-axis"><span>Jun 9</span><!-- one per day, or thin out on long ranges --></div>
   ```
   Then the accordion, one `<details class="ws-item">` per workstream (first one `open`):
   ```html
   <div class="ws-accordion">
     <details class="ws-item" open>
       <summary>
         <span class="ws-item-swatch" style="background:var(--ws-1);"></span>
         <span class="ws-item-name">Content corpus</span>
         <span class="ws-item-cost">~$820</span>
         <span class="ws-item-caret">›</span>
       </summary>
       <div class="ws-item-body">
         <p class="ws-item-blurb">What this workstream covers, in one or two sentences.</p>
         <div class="ws-item-stat">~$820 · ~32h · ~85 entries · active 20 of 25 days</div>
       </div>
     </details>
   </div>
   ```
   Compute the hours figure the same way `generate-recap.mjs` computes a day's hours —
   summing each classified entry's own start-to-end span — bucketed by workstream instead
   of by day. Include it by default, not only when a cost-redacted version is requested:
   it's a genuinely useful number on its own, and having it already present means producing
   the cost-redacted version later is just deleting the `~$820 ·` clause from this same line
   rather than deriving anything fresh (see "Cost-redacted version" below).
   Use the `--ws-1` through `--ws-6` CSS variables already defined in the template for up
   to 6 workstreams; pick additional hex colors in the same desaturated warm-palette family
   if there are more.

   Wire up hover/click with a small inline `<script>` right after the section markup,
   reusing the base template's already-global `openPanel()` and `dayKeys` — don't
   reinvent panel-opening logic. **The hover handler must update every row's
   `.ws-gantt-pct` from that day's `data-pct`, not just the info line above the
   chart** — a hover interaction that only updates the date and not the numbers beside
   it is half-built and reads as broken:
   ```html
   <script>
   (function () {
     var section = document.currentScript.closest('[data-section="workstreams"]');
     var hits = section.querySelectorAll('.ws-gantt-hit');
     var info = section.querySelector('.hover-info');
     var defaultInfo = info ? info.textContent : '';
     var rows = section.querySelectorAll('.ws-gantt-row');
     // one .ws-gantt-pct + its NodeList of .ws-gantt-cell (in day order) per row, same order as `hits`
     var rowData = Array.prototype.map.call(rows, function (row) {
       return { pctEl: row.querySelector('.ws-gantt-pct'), cells: row.querySelectorAll('.ws-gantt-cell') };
     });
     hits.forEach(function (hit, dayIdx) {
       hit.addEventListener('mouseenter', function () {
         hits.forEach(function (h) { h.classList.remove('hover'); });
         hit.classList.add('hover');
         if (info) {
           var day = hit.getAttribute('data-day');
           var d = (typeof dailyData !== 'undefined') ? dailyData.find(function (x) { return x.key === day; }) : null;
           var meta = d ? ' &mdash; ~$' + d.c + ' &middot; ' + d.n + ' entries' : '';
           info.innerHTML = '<strong>' + day + '</strong>' + meta + ' &middot; <span class="view-link">view day &rarr;</span>';
         }
         rowData.forEach(function (row) {
           var cell = row.cells[dayIdx];
           if (cell && row.pctEl) row.pctEl.textContent = cell.getAttribute('data-pct') + '%';
         });
       });
       hit.addEventListener('mouseleave', function () {
         hit.classList.remove('hover');
         if (info) info.textContent = defaultInfo;
         rowData.forEach(function (row) { if (row.pctEl) row.pctEl.textContent = row.pctEl.getAttribute('data-default') + '%'; });
       });
       hit.addEventListener('click', function (ev) {
         ev.preventDefault();
         openPanel(hit.getAttribute('data-day'));
       });
     });
   })();
   </script>
   ```
   The no-JS fallback still works without this script: each `.ws-gantt-hit` is a real
   `<a href="#panel-...">` pointing at the same day panels the daily-activity chart uses;
   the percentages just stay at their default share without JS.

5. **Enrich the day panels with the workstream mix.** This is not optional cosmetic
   polish — the design this skill follows always shows it, and a day panel without it is
   an incomplete implementation. The template already has an empty
   `#panel-ws-mix`/`#panel-ws-mix-bar`/`#panel-ws-mix-legend` block (hidden by default)
   between the panel header and the entry list, and `openPanel()` already renders it
   whenever a global `wsMix` object is present. Define that object as part of this step —
   one inline `<script>` tag, `var wsMix = { "<day key>": [{name, pct, color}, ...], ... }`,
   keyed with the exact same day keys `E`/`dailyData` already use. Then, for the no-JS
   static fallback, insert the equivalent markup directly into each already-generated
   `<div class="day-panel-target">` block, right after its `.panel-header` and before
   `.panel-body`:
   ```html
   <div class="panel-ws-mix">
     <div class="panel-ws-mix-label">Workstream mix</div>
     <div class="panel-ws-mix-bar">
       <div style="flex:20 0 0;background:var(--ws-1);"></div>
       <!-- one div per active workstream that day, flex value = its share (not required to sum to 100) -->
     </div>
     <div class="panel-ws-mix-legend">
       <span class="panel-ws-mix-item"><span class="panel-ws-mix-swatch" style="background:var(--ws-1);"></span>Content corpus · 20%</span>
       <!-- one span per active workstream that day, same order as the bar segments -->
     </div>
   </div>
   ```
   Skip a day only if it has no logged activity at all.

**Editing an existing recap's Workstreams section — check for dead code first.** If a
prior version of this section used the older positioned-swimlane markup (`#ws-container`,
`#ws-click-zones`, `#ws-hover-line`, functions like `getDayFromX`/`showHoverLine`), grep the
page's closing `<script>` block for those identifiers before assuming the new gantt markup
is enough on its own. A leftover `document.getElementById('ws-click-zones').addEventListener(...)`
against an element that no longer exists throws on page load and **silently kills every
statement after it in that script tag** — including the side panel's own setup further
down, which is why `openPanel()` can look correct in isolation while every click on the
page does nothing. Delete stale swimlane-era code entirely rather than leaving it inert;
check the console for errors after any Workstreams edit, not just the visual result.

#### Milestones & capabilities timeline

Generate a dual timeline using the `.tl-row` structure (see the template's CSS — colors
are already themed via `--ms-*`/`--cap-*` variables, just use the classes). **The card
shape (border-radius, padding, width) lives on the base `.tl-card` class; `.tl-ms-card` /
`.tl-cap-card` only add the color** — a card needs both classes together, or it renders as
a bare bordered rectangle with no rounded corners or padding:
- Left column: `.tl-left` wrapper containing `.tl-card.tl-ms-card` (title `.tl-ms-title`, desc `.tl-ms-desc`)
- Center: `.tl-date-badge` inside `.tl-pill-cell`
- Right column: `.tl-right` wrapper containing `.tl-card.tl-cap-card` (title `.tl-cap-title`, desc `.tl-cap-desc`)
- Gap rows: `.tl-row.tl-gap-row` with `.tl-gap-dot` in the center and `.tl-gap-l`/`.tl-gap-r` text either side
- Wrap the whole thing in `.tl-wrap` with a `.tl-rail` div for the vertical center line, and
  a `.tl-heads` row (`.tl-header-left` "Milestones" / `.tl-header-right` "Capabilities") above it
- **Always include `.tl-mobile-legend`** right after `.tl-heads` — two swatch+label spans,
  `.tl-legend-item` wrapping a `.tl-legend-swatch.tl-legend-ms` + "Milestone" and a
  `.tl-legend-item` wrapping a `.tl-legend-swatch.tl-legend-cap` + "Capability". It's hidden
  by the template's CSS above the mobile breakpoint (`.tl-heads` already names the two
  columns there) and shown in place of `.tl-heads` below it, once the header disappears and
  rows stack into one column — without it, color is the only thing telling a milestone card
  from a capability card on a phone, which isn't enough on its own. Never omit it as
  "redundant with the header" — the two are mutually exclusive by breakpoint, not duplicates.

```html
<div class="tl-wrap">

  </div>
  <div class="tl-row">
    <div class="tl-left">
      <div class="tl-card tl-ms-card">

      </div>
    </div>
    <div class="tl-pill-cell"><span class="tl-date-badge">Jun 10</span></div>
    <div class="tl-right">
      <div class="tl-card tl-cap-card">

```

#### How you use Claude (patterns)

Analyze the log for 3–6 recurring patterns in how the user works with Claude, ordered
most distinctive first. Each pattern is a `.pattern-block` card — same visual weight and
card treatment as a `.finding` in the Findings section below, so the two AI sections read
as one consistent system rather than two different styles. Internal structure:

1. `.pattern-head-row` — a flex row with `.pattern-head` (the claim as a headline,
   "Claude as critic, not just builder", not a topic label — flex:1, sits on the left)
   and `.pattern-tag` (the category chip, sits on the right), mirroring `.finding-head`'s
   title-then-tag layout exactly:
   - `.tag-dynamics` (purple): collaboration patterns
   - `.tag-forward` (green): emerging practices
   - `.tag-optimize` (pink): optimization opportunities (use when a recommendation
     builds on the pattern)
2. One narrative paragraph, **2–4 sentences**, describing the pattern itself — what it
   is and why it works or matters. Lead with the description, not the tally. A count or
   date is supporting color, not the opening move — fold one in only where it sharpens
   the claim ("about a fifth of entries," "roughly every three days"), and skip it
   entirely when the pattern is better made by description alone. Don't manufacture a
   `.pattern-evidence` line as a reflex on every block; use it only when a specific
   number materially strengthens the claim, and vary whether it's even present.
3. An optional verbatim prompt quote in a `.pattern-quote` blockquote with a `.q-ref`
   attribution (entry # if the log numbers entries, else the date) — include one **only**
   where it's genuinely the clearest way to make the point (a distinctive phrase, a
   moment the user diagnosed the pattern themselves), not as a default per block. Most
   patterns should land without one. When used, quote verbatim — elide long prompts with
   … — never paraphrase; preserving exact wording is the point of the log, and the recap
   is private unless the user opts to share it.

Vary the mix across the section — some blocks are description-only, some add one
sharp number, a couple earn a quote. Uniform structure across every block is a sign
the density rule above was ignored — but that variation lives in the *content*, not the
card styling; every block still gets the same card, same head-row layout.

```html
<div class="pattern-block">
  <div class="pattern-head-row">
    <span class="pattern-head">Claude as critic, not just builder</span>
    <span class="pattern-tag tag-dynamics">Collaboration</span>
  </div>
  <p>A recurring share of the work is Claude reviewing, auditing, or testing rather than building — adversarial spec reviews, regression passes, a closing multi-agent audit. The clearest instance: the user wrote a spec with Claude, then asked Claude to attack it, then folded the findings back in before building.</p>
  <blockquote class="pattern-quote">"Update both pipeline specs with every decision from the adversarial review…"<span class="q-ref">— Jun 22</span></blockquote>
</div>
```

#### Findings, ranked by value

Actionable, ranked, and specific to *this* project — never generic advice that could apply
to any project. One flat list ordered by impact; the ranking IS the structure (don't group
by category and rank within groups). Aim for a handful — **roughly 3–6**. This is a ranked
top-N: a candidate earns a slot only by out-ranking the others, never by belonging to a
category. There is no coverage target and no per-category minimum — a strong recap might be
four findings from one lens, or skip whole lenses entirely.

**Lenses to consider — invisible scaffolding, not sections.** Run the log past each and see
whether *this* project throws off a finding that clears the inclusion bar below. Most lenses
won't fire on any given project; that's expected, not a gap. The reader never sees these lens
names — only a ranked list with a fitting tag per card.

- **Direction — where the project should go next.** Its current maturity implies a next move:
  "live but young → reliability, not features"; "feature-complete but unproven with real
  users → a review round beats more features"; "broad but shallow → depth over coverage."
- **Leverage — reusable structure hiding in repetition.** A multi-step workflow repeated
  often enough to codify (skill, command, hook, template, checklist), or a Claude / Claude
  Code capability the log shows *would have helped a specific moment*. Only when the
  repetition or the missed fit is real — not one-offs, not "you could try X."
- **Efficiency — the same result for less.** Where spend or effort concentrated and a clear,
  material change would move it: a lopsided model mix (near-total Opus while a big share of
  the work was mechanical), a task class that could run cheaper, or visible thrash a
  spec-first / test-first move would have prevented.
- **Durability — what bites later.** Built-but-unverified surfaces, a bug *class* that
  recurred (guardrail territory), a single point of failure, or decision/requirements debt.
- **Knowledge — compounding value.** Rules the project learned the hard way and the *lag*
  before they were codified (= rework), or a technique invented in one corner that
  generalizes to other loops in the same project.
- **Collaboration — how you prompt and work with Claude.** A specific, recurring, material
  cost in the collaboration itself: asks that needed several clarification rounds because the
  prompt under-specified the output; context re-established from scratch each session that a
  `CLAUDE.md` note would carry; a build-then-audit habit where a spec-first pass would have
  caught it pre-build. Held to the same bar — only when it's stark and specific, **never**
  generic "be more specific." A pattern *describes* how they work; a Collaboration finding
  *changes* it (see boundary below).

**The inclusion bar — what earns a slot.** A finding surfaces only when all three hold:
1. **Material, not marginal.** The signal is stark, not a hairline call. ~100% Opus is a
   finding; "~3% cheaper if you'd swapped two calls" is noise. A lens fires only when the
   project's real data crosses from ambient into stark.
2. **Specific to this project** — grounded in what actually happened here, not restatable for
   any project.
3. **Changes what they'd do or know.** "You use Opus" when Opus was deliberate isn't insight;
   "near-total Opus, and here's the ~40% of the work mechanical enough to run cheaper" is. If
   the reader would shrug "I know," it's not a finding.

**Rank by impact — and the #1 slot is the most valuable *still-actionable* finding, not the
scariest headline.** Don't lead with an already-resolved issue whose fix is "just note it"; a
top finding the reader can't act on is a mis-rank. And don't demote a forward-looking or
interpretive finding for lacking a tally — a number sharpens a finding where one exists, but
its absence doesn't lower its rank. Weighing findings by how countable they are rather than
how much they matter is what buries Direction, Durability, Knowledge, and Collaboration under
retrospective process-hygiene.

**Every Fix is a forward action the reader can still take.** If an issue is already fixed,
either drop it or state the *transferable lesson* as the action ("initialize git on day one
of the next project") — never ship a finding whose Fix amounts to "nothing to do."

**Titles are plain-language stakes, not the mechanism.** Someone who isn't in the code should
get what went wrong and why it matters from the title alone. Keep file names, class names,
and jargon ("optimistic-UI-vs-disk desync", port numbers, `writeBack.ts`) **out** of the
title and in the evidence paragraph, and don't preview the body in the title — the title is
the hook, the paragraph is the detail. E.g. "You kept re-fixing the same broken preview
instead of pinning it once", not "Preview harness port-mismatch recurred across 11 chunks".

**Scrub PII.** Prompts are verbatim and the recap may be shared. Don't surface personal data
that isn't load-bearing — emails, phone numbers, names of third parties — even when it sits
right there in the evidence. "A reviewer", not their address.

**Boundary with "How you use Claude."** Descriptive "here's how you work" is a pattern; a
specific, material thing to *change* is a finding. If an observation could go either way,
prefer stating the actionable change as a Collaboration finding here. Never state the same
thing in both sections.

Each finding is a `.finding` div, numbered `01`, `02`, `03`… The single most impactful
finding gets `.finding.finding-top` (the emphasized gradient card) and a `.finding-tag.solid`
chip reading its urgency (e.g. "FIX TODAY", "KEY FINDING"). Every other finding uses the plain
`.finding` card with a `.finding-tag.muted` chip stating its category. Color the **finding
number** (`.finding-num`) to match the chip color so a skimming reader can pattern-match down
the left edge:

| Chip | Color | Lens it usually carries |
|---|---|---|
| Fix today / Key finding (solid, top card only) | `var(--accent)` | the #1, any lens |
| Direction | `var(--cat-blue)` | Direction |
| Automation | `var(--cat-purple)` | Leverage |

| Knowledge | `var(--cat-teal)` | Knowledge |

| Workflow | `var(--cat-green)` | process-friction / recurring rework |
| Housekeeping | `var(--cat-gray)` | minor hygiene |

Structure of one finding:
1. `.finding-head` — `.finding-num` ("01"), `.finding-title` (the headline, stated as a
   claim: "Your automation layer has been dead for a week", not a topic label), `.finding-tag`.
2. One paragraph of evidence/context, numbers in `<strong>`, `<code>` for file/command names.
   Where a dollar figure is the natural evidence, cite the matching time figure in the same
   sentence too ("...burned $340 across 12 requests over 6 hours...") — by default, not only
   for a cost-redacted version. Same reasoning as the Workstreams stat line above: the time
   figure is useful on its own, and having it already there means the cost-redacted version
   is just dropping the dollar clause, not rewriting the evidence (see "Cost-redacted
   version" below).
3. `.finding-rule` divider, then `.finding-fix` — `<b>Fix:</b>` with the concrete first
   action, named specifically enough to start on.
4. Optional `<blockquote>` with `.fq-text` and `.fq-ref` — only when a verbatim quote is
   genuinely the clearest way to make the point (same restraint as the patterns section).

The closing **"If you only do three things this week"** capstone is generated together
with Findings — it's not a separate picker item. Pick the top three findings across the
whole ranked list and render them as `.capstone-row`s inside `.capstone-table` (with one
`.capstone-header-row` above): `.col-n` (rank), `.col-action`, `.col-payoff` (what the
reader gets, not what they do). Write it so it works as the only thing a skimming reader
takes away — replace the template's `{{THREE_THINGS_SECTION}}` placeholder the same way
`{{FINDINGS_SECTION}}` gets replaced.

Between the findings list and the capstone the template carries a static `.findings-more`
note — a "these are the top few; want the full list? ask Claude like this" affordance with a
copyable example prompt. It's boilerplate, identical every recap, so it lives in the template,
not here — leave it as rendered; don't author, remove, or rewrite it.

### Step 4: Remove unselected sections

For any section the user didn't select, remove the entire `<section>` element from the HTML
(find by `data-section` attribute). Don't leave empty sections. Two `<section>` elements
share `data-section="daily-activity-cost"` (Daily activity and Cost & time) — since they're
never selected independently (Step 1), removing or keeping one by that attribute always
means removing or keeping both.

### Step 5: Deliver

Tell the user where the dated recap file (`session-log/YYYY-MM-DD-Recap.html`) is, offer to
open it, and mention — in the same message, not a separate question — that the dollar
figures can be swapped out for their time/percentage equivalent if they'd rather not
disclose them (say, if this is headed to someone else). Something like: "saved to
`session-log/2026-07-06-Recap.html` — if you don't want to show what this cost, I can make
a version with the dollar figures swapped for time instead." Never build that version
unless they actually ask for it — it's an offer, not a default. And don't read a generic
"make this shareable" or "something I can send someone" as *implying* the cost swap; those
could mean anything (trimming prompts, cutting sections, nothing at all). This methodology
is specifically for when someone asks to remove or swap out the dollar amounts themselves.

## Cost-redacted version

When the user specifically asks to remove or swap out the dollar figures — "take out the
dollar amounts," "swap cost for hours," "I don't want to show what this cost" — or takes
you up on the offer from Step 5 above, build a second file, never overwriting the private
one:

1. Run `node _scripts/generate-recap.mjs --share` (add `--root <dir>` as usual). This writes
   `session-log/YYYY-MM-DD-Recap-Shared.html` with the deterministic base already converted:
   stat row and daily bars in hours instead of dollars (peak day recomputed by hours, since
   the highest-cost day and the highest-hours day aren't guaranteed to be the same one), cost
   by model shown as percentage only, cost by window shown as hours by window, and the day
   panels' cost line swapped for hours. The underlying data embedded in the page (not just
   the visible labels) omits cost entirely in this mode — there's nothing to recover by
   viewing source.
2. Fill in the AI sections by **adapting** what you already wrote for the private recap of
   the same date, not re-deriving anything from scratch — the analysis is identical, only
   the cost-bearing details change:
   - **Workstreams**: the gantt chart and percentages are entry-count-based and don't involve
     cost at all, so copy that part verbatim. In the accordion's stat line, drop the
     `~$820 ·` clause and keep the rest (`~32h · ~85 entries · active 20 of 25 days`) — it's
     already there from Step 3 above, so this is deletion, not computation.
   - **Milestones & capabilities timeline**: copy verbatim; nothing there should reference cost.
   - **How you use Claude (patterns)**: copy verbatim, but scan for any incidental dollar
     mention in a pattern's evidence and drop it if present — same discipline as Findings,
     just less likely to come up.
   - **Findings**: copy each finding, and where its evidence sentence cites a dollar figure
     (it should, per Step 3's Findings instructions above), drop that clause and keep the
     time clause that's already sitting right next to it.
3. Tell the user where the shared file is. Don't proactively offer it a second time in the
   same conversation once it's built.
